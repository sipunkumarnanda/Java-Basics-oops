# 🌟 Introduction

In **Chapter 1**, we learned that **Abstraction** is the process of **hiding implementation details while exposing only the essential functionality**.

We also learned that the outside world should interact with an **abstract class** or an **interface**, while the implementation remains inside the child classes.

But a very important question still remains.

> **Why do we even need abstraction?**

Why can't we simply write normal classes and create objects directly?

```java
SavingsAccount account = new SavingsAccount();
```

Why introduce concepts like:

* Abstract Classes
* Interfaces
* Contracts
* Polymorphism

Wouldn't programming be much simpler without them?

The answer is:

> **Programming is simple for small applications, but software engineering is about building applications that can grow, change, and be maintained for years.**

Abstraction was introduced to solve the problems that appear when software becomes large.

---

# 🏗️ Imagine Building a Small Application

Suppose you are learning Java.

Your teacher asks you to create a simple banking program.

You write:

```java
class SavingsAccount {

    void deposit(double amount) {
        System.out.println("Money Deposited");
    }

    void withdraw(double amount) {
        System.out.println("Money Withdrawn");
    }

}
```

Then,

```java
public class Main {

    public static void main(String[] args) {

        SavingsAccount account = new SavingsAccount();

        account.deposit(5000);

    }

}
```

Everything works perfectly.

There is only one account type.

Only one programmer is working.

Only one file exists.

Everything looks simple.

# 🚨 Problem 1: No Common Contract

Imagine three developers are working on the same banking application.

Each developer is responsible for one account type.

Developer A creates:

```java
class SavingsAccount {

    void depositMoney(double amount){

    }

}
```

Developer B creates:

```java
class CurrentAccount {

    void addMoney(double amount){

    }

}
```

Developer C creates:

```java
class SalaryAccount {

    void credit(double amount){

    }

}
```

All three methods perform the same task.

They deposit money into the account.

But look at the method names.

```text
depositMoney()

addMoney()

credit()
```

Every developer used a different name.

---

# 😨 Why Is This a Problem?

Now another developer joins the company.

He wants to deposit money.

He opens the project.

What does he see?

```text
SavingsAccount

↓

depositMoney()

------------------------

CurrentAccount

↓

addMoney()

------------------------

SalaryAccount

↓

credit()
```

He becomes confused.

Questions arise immediately.

* Which method should I call?

* Why are all names different?

* Are they doing different work?

* Which one is the company standard?

The project becomes inconsistent.

# 📉 As the Project Grows...

Imagine instead of three account types,

there are fifty.

```text
SavingsAccount

CurrentAccount

SalaryAccount

BusinessAccount

StudentAccount

LoanAccount

PremiumAccount

CorporateAccount

...

50 Different Classes
```

---

# 🎯 How Abstraction Solves This Problem

Instead of allowing every developer to invent their own methods,

the company creates a common contract.

```java
abstract class BankAccount{

    abstract void deposit(double amount);

    abstract void withdraw(double amount);

}
```

Notice something important.

The abstract class does **not** explain how money is deposited.

It simply defines the operations that every bank account must support.

It says:

> "If you are a BankAccount, you must provide these methods."

This is called a **contract**.

---

# 🧩 What Is a Contract?

A contract is an agreement.

In Java,

an abstract class acts like an agreement between the parent class and all child classes.

It says:

```text
Every BankAccount

↓

Must Deposit Money

↓

Must Withdraw Money
```

The parent class does **not** care how each account performs these operations.

It only guarantees that these operations exist.

---

# ✅ Child Classes Follow the Contract

Savings Account

```java
class SavingsAccount extends BankAccount{

    @Override
    void deposit(double amount){

        System.out.println("Savings Deposit");

    }

    @Override
    void withdraw(double amount){

        System.out.println("Savings Withdrawal");

    }

}
```

Current Account

```java
class CurrentAccount extends BankAccount{

    @Override
    void deposit(double amount){

        System.out.println("Current Deposit");

    }

    @Override
    void withdraw(double amount){

        System.out.println("Current Withdrawal");

    }

}
```

Now every account follows the same contract.

Every programmer already knows that every `BankAccount` has:

```text
deposit()

withdraw()
```

No guessing.

No confusion.

No inconsistent method names.

# 🌍 Real-Life Analogy: Electrical Plug Standards

Imagine every mobile company designed its own wall socket.

Apple:

```text
🔌 Type A
```

Samsung:

```text
🔌 Type B
```

OnePlus:

```text
🔌 Type C
```

Google:

```text
🔌 Type D
```

Every house would need four different sockets.

Charging devices would become frustrating.

Instead, countries define a **standard electrical socket**.

Every manufacturer follows that standard.

The socket acts like a **contract**.

The phone manufacturer is free to design the charger internally, but the plug must match the agreed standard.

Similarly in Java:

* `BankAccount` is the standard (contract).
* `SavingsAccount`, `CurrentAccount`, and `SalaryAccount` are different implementations of that contract.

---

# 💡 Key Takeaways

✅ Abstraction was introduced because software changes continuously.

✅ Large applications require common rules for all developers.

✅ An abstract class defines **what operations must exist**, not how they are implemented.

✅ A common contract makes projects easier to understand, maintain, and extend.

✅ Abstraction's first major benefit is **standardization**, not code reuse.

# 🎤 Interview Questions

### Q1. Why was abstraction introduced?

**Answer:**
Abstraction was introduced to simplify software design by defining common contracts, reducing dependency on implementation details, improving maintainability, and allowing software to evolve without affecting client code.

---

### Q2. What is a contract in Java?

**Answer:**
A contract is a set of methods that every implementing class must provide. An abstract class or interface defines this contract without specifying the complete implementation.

---

### Q3. Does abstraction automatically remove duplicate code?

**Answer:**
No. Abstraction defines a common contract. Duplicate code is reduced only when common implementation is shared through inheritance or other design techniques.
