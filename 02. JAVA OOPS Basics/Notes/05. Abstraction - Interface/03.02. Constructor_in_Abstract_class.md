# 🏗️ Constructors in Abstract Classes | JVM Object Creation | Memory Flow

# 🌟 Introduction

In the previous part, we learned that:

✅ An abstract class cannot be instantiated.

For example,

```java
abstract class BankAccount{

    abstract void deposit(double amount);

}
```

The following code is illegal.

```java
BankAccount account = new BankAccount();
```

The compiler reports an error because `BankAccount` is abstract.

Now a very interesting question arises.

> **If we cannot create an object of an abstract class, then why does Java allow constructors inside an abstract class?**

This is one of the most frequently asked Java interview questions.

To answer it, we first need to understand **how Java creates an object**.

---

# 🤔 How Does Java Create an Object?

Suppose we have a simple class.

```java
class Student{

    Student(){

        System.out.println("Student Constructor");

    }

}
```

Object creation:

```java
Student s = new Student();
```

What happens internally?

```text
new Student()

        │

        ▼

Memory Allocated

        │

        ▼

Constructor Executed

        │

        ▼

Object Ready
```

This process is straightforward.

---

# 🏛️ What About Inheritance?

Now consider inheritance.

```java
class Person{

    Person(){

        System.out.println("Person Constructor");

    }

}

class Student extends Person{

    Student(){

        System.out.println("Student Constructor");

    }

}
```

Now create an object.

```java
Student s = new Student();
```

Many beginners think:

```text
Student Constructor

↓

Object Created
```

But this is **not** what happens.

---

# 🔍 Actual Execution Flow

When Java creates a child object,

it **always initializes the parent first**.

Execution order:

```text
new Student()

        │

        ▼

Allocate Memory

        │

        ▼

Call Parent Constructor

        │

        ▼

Initialize Parent State

        │

        ▼

Call Child Constructor

        │

        ▼

Initialize Child State

        │

        ▼

Object Ready
```

Output:

```text
Person Constructor

Student Constructor
```

---

# ❓ Why Does Java Call the Parent Constructor First?

Imagine this class.

```java
class Person{

    String name;

}
```

Student inherits Person.

```java
class Student extends Person{

    int rollNumber;

}
```

When creating a Student object,

the object contains both

```text
Person Data

+

Student Data
```

Visualize it.

```text
Student Object

----------------------

name

rollNumber

----------------------
```

Since the child object also contains the parent's fields,

Java must initialize the parent part first.

Only then can it initialize the child.

This is why the parent constructor executes first.

---

# 🏛️ Now Replace Person with an Abstract Class

```java
abstract class BankAccount{

    BankAccount(){

        System.out.println("BankAccount Constructor");

    }

}
```

Child class:

```java
class SavingsAccount extends BankAccount{

    SavingsAccount(){

        System.out.println("Savings Constructor");

    }

}
```

Object creation:

```java
SavingsAccount account =
        new SavingsAccount();
```

Output:

```text
BankAccount Constructor

Savings Constructor
```

Notice something surprising.

Even though

```java
new BankAccount();
```

is illegal,

its constructor still executes.

Why?

---

# 🎯 The Most Important Rule

Java **never creates a BankAccount object**.

Instead,

Java creates a **SavingsAccount object**.

During the creation of that object,

Java initializes the BankAccount portion first.

Visualize the object.

```text
SavingsAccount Object

----------------------------------

BankAccount Variables

----------------------

SavingsAccount Variables

----------------------------------
```

The BankAccount constructor initializes only the BankAccount part.

Then the SavingsAccount constructor initializes the remaining part.

Finally,

the complete SavingsAccount object is ready.

---

# 🌍 Real-Life Analogy

Imagine constructing a house.

First,

workers build the

```text
Foundation
```

Then they build the

```text
Ground Floor
```

Finally,

they build the

```text
First Floor
```

Would they build the first floor before the foundation?

❌ No.

The foundation must exist first.

Similarly,

the abstract class acts like the foundation.

The child class completes the building.

---

# 🔍 Does Java Ever Create a Parent Object?

This is a common misconception.

Many beginners think

```text
BankAccount Object

↓

SavingsAccount Object
```

Two objects are created.

❌ Wrong.

Only **one object** is created.

That object is

```text
SavingsAccount
```

Inside that object,

Java initializes the BankAccount part first.

---

# 📊 Memory Representation

Suppose

```java
abstract class BankAccount{

    String accountNumber;

}
```

```java
class SavingsAccount extends BankAccount{

    double interestRate;

}
```

Object:

```java
SavingsAccount account =
        new SavingsAccount();
```

Memory:

```text
Heap Memory

----------------------------------

SavingsAccount Object

----------------------------------

accountNumber

interestRate

----------------------------------
```

There is **not** a separate BankAccount object.

The parent fields become part of the child object.

---

# 🔄 Constructor Chaining

Whenever a child constructor executes,

Java automatically inserts

```java
super();
```

Example:

```java
class SavingsAccount extends BankAccount{

    SavingsAccount(){

    }

}
```

Compiler changes it to

```java
class SavingsAccount extends BankAccount{

    SavingsAccount(){

        super();

    }

}
```

The call to `super()` invokes the constructor of the abstract class.

This happens even though the abstract class itself cannot be instantiated.

---

# 🚨 Why Constructors Are Allowed in Abstract Classes

An abstract class may contain:

```java
String accountNumber;

double balance;

Date createdAt;
```

These fields need initialization.

The constructor is responsible for initializing this common state.

Example:

```java
abstract class BankAccount{

    String accountNumber;

    BankAccount(String number){

        accountNumber = number;

    }

}
```

Child:

```java
class SavingsAccount extends BankAccount{

    SavingsAccount(){

        super("SB1001");

    }

}
```

Now every child automatically receives the initialized parent data.

---

# 💡 Key Insight

An abstract class constructor is **not** used to create an abstract-class object.

It is used to initialize the **parent portion of the child object**.

This is why Java allows constructors in abstract classes.

---

# 🎯 Interview Questions

### Q1. Can an abstract class have a constructor?

**Answer:**

Yes. An abstract class can have constructors. They are executed when a concrete subclass object is created.

---

### Q2. Why is the constructor of an abstract class executed even though the class cannot be instantiated?

**Answer:**

Because the constructor initializes the parent part of the child object. Java creates only the child object, but before initializing the child-specific fields, it initializes the inherited fields by calling the abstract class constructor.

---

### Q3. Does Java create an object of the abstract class when a subclass object is created?

**Answer:**

No. Java creates only one object—the concrete subclass object. The abstract class constructor initializes the inherited portion of that object; it does not create a separate abstract-class instance.

---

# 🧠 Interview Tip

If an interviewer asks:

> **"Why does an abstract class have a constructor if it cannot be instantiated?"**

Don't answer:

> "Because Java allows it."

Instead explain the lifecycle:

> "When a concrete subclass object is created, Java first executes the abstract class constructor to initialize the inherited state. The constructor belongs to the initialization process of the child object, not to the creation of an abstract-class object."

This demonstrates a deeper understanding of Java's object creation process.
