

# 📘 Chapter: Reference Types, Reference Variables & Objects (Part 1)

---

# 🌟 Introduction

One of the biggest reasons beginners struggle with **Inheritance**, **Polymorphism**, **Abstraction**, **Coupling**, **Dependency Injection**, and even **Spring Framework** is because they don't fully understand one fundamental concept:

> **References and Objects**

Consider the following Java statement.

```java
BankAccount account = new SavingsAccount();
```

Most beginners think this statement creates only one thing.

However, this single line actually involves **three different concepts**.

```text
BankAccount      account      =      new SavingsAccount();
     │               │                     │
     │               │                     │
     │               │                     └── Object (Object Type)
     │               └──────────────────────── Reference Variable
     └──────────────────────────────────────── Reference Type
```

If you understand these three concepts, many advanced Java topics become much easier because they are all built on this foundation.

By the end of this chapter, you will clearly understand:

* 📦 What an object really is
* 📍 What a reference is
* 📝 What a reference variable is
* 🏷️ What a reference type is
* 🎯 What an object type is
* 🔍 Why Java separates the reference type from the object type

---

# 🏗️ Class vs Object

Before understanding references, we must first understand the difference between a **class** and an **object**.

Many beginners think they are the same thing.

They are not.

---

## 📖 What is a Class?

### Definition

> A class is a blueprint or template that defines the properties (data) and behaviors (methods) of an object.

Example

```java
class Car {

    String brand;

    void start() {

        System.out.println("Car Started");

    }

}
```

This code only describes what a Car should look like.

It says:

Every car has

* a brand

and every car can

* start()

But notice something.

There is still **no actual car**.

The class is only a design.

---

## 🏠 Real-Life Analogy

Imagine an architect designing a house.

Blueprint

```text
Bedroom

Kitchen

Garage

Bathroom
```

Can someone live inside the blueprint?

No.

The blueprint only describes how the house should be built.

Similarly,

```text
Class

↓

Blueprint
```

---

## 📖 What is an Object?

### Definition

> An object is a real instance of a class that occupies memory and contains actual data and behavior.

To create an object,

we use the `new` keyword.

```java
Car car = new Car();
```

Now an actual Car object exists in memory.

Unlike the class,

the object is real.

```text
Class

↓

Blueprint



Object

↓

Actual House
```

---

# 💾 What Does the `new` Keyword Actually Do?

Many beginners think

```java
new Car();
```

simply creates an object.

That is only partially correct.

Let's understand exactly what happens.

Suppose memory is empty.

```text
Memory

(empty)
```

Now Java executes

```java
new Car();
```

Java creates a new Car object somewhere in memory.

```text
Memory

+---------------------------+
| Car Object                |
|---------------------------|
| brand = null              |
|                           |
| start()                   |
+---------------------------+
```

Now an object exists.

But there is a problem.

How do we access it?

Java knows where the object is.

You don't.

You need something that remembers its location.

That "something" is called a **reference**.

---

# 📍 What is a Reference?

## 📖 Definition

> A reference is a value that points to the memory location of an object.

Notice carefully.

The reference is **not** the object.

It only stores where the object is located.

Think of it as the address of a house.

---

### 🏠 House Analogy

Imagine your friend asks,

"Where do you live?"

Do you hand over your house?

No.

You give the address.

```text
221B Baker Street
```

The address is not the house.

It only tells people where the house exists.

Similarly,

```text
Reference

↓

Address



Object

↓

House
```

The reference helps Java locate the object.

---

# 📝 What is a Reference Variable?

## 📖 Definition

> A reference variable is a variable that stores the reference (memory address) of an object.

Consider

```java
Car car = new Car();
```

Let's identify the parts.

```text
Car      car      =      new Car();
│         │                 │
│         │                 └── Object
│         └──────────────────── Reference Variable
└────────────────────────────── Reference Type
```

Here,

```java
car
```

is the **reference variable**.

Its job is simple.

It stores the reference returned by

```java
new Car()
```

Visual representation

```text
Reference Variable

car

↓

Car Object
```

Notice something important.

The variable

```java
car
```

does **not** store the object.

It stores only the reference that points to the object.

---

## 💡 Common Beginner Misconception

Many beginners think

```java
Car car;
```

creates a Car object.

It does not.

It only creates a variable capable of storing the reference to a Car object.

No object exists yet.

The object is created only when Java executes

```java
new Car();
```

---

# 🏷️ What is a Reference Type?

## 📖 Definition

> The reference type is the declared type of the reference variable. It determines what kind of objects the variable can refer to and what methods the compiler allows you to access.

Consider

```java
Car car = new Car();
```

Reference Type

```java
Car
```

Reference Variable

```java
car
```

Object Type

```java
Car
```

Now consider a more interesting example.

```java
Animal animal = new Dog();
```

Break it down.

```text
Animal      animal      =      new Dog();
   │             │                 │
   │             │                 │
   │             │                 └── Object Type
   │             └──────────────────── Reference Variable
   └────────────────────────────────── Reference Type
```

The reference type is

```java
Animal
```

This tells Java

> "The variable `animal` can refer to any object that is an Animal."

That includes

```java
animal = new Dog();
animal = new Cat();
animal = new Lion();
```

because each one is an Animal.

But this is not allowed.

```java
animal = new Car(); // ❌
```

because a Car is not an Animal.

---

# 🎯 What is Object Type?

## 📖 Definition

> The object type is the actual class whose object is created using the `new` keyword.

Example

```java
Animal animal = new Dog();
```

Reference Type

```text
Animal
```

Reference Variable

```text
animal
```

Object Type

```text
Dog
```

Visual

```text
animal
   │
   ▼
Dog Object
```

Notice something very important.

The variable is declared as

```java
Animal
```

but the actual object belongs to

```java
Dog
```

The object type is determined only by

```java
new Dog()
```

The reference type does **not** determine which object is created.

It only determines what kinds of objects the variable is allowed to reference.

---

Imagine a parking lot.

A parking slot is labeled

```text
Vehicle
```

This is like the **reference type**.

Today,

a Car is parked.

Tomorrow,

a Bike is parked.

The parking slot label never changes.

Only the vehicle changes.

```text

↓

Car
```

Later,

```text
Vehicle

↓

Bike
```

Similarly,

```java
Vehicle vehicle = new Car();
```

Later,

```java

```

The reference type remains

```java
Vehicle
```

Only the object changes.

# 🔍 Breaking Down

```java
BankAccount account = new SavingsAccount();
```

Let's identify every part.

```text
BankAccount      account      =      new SavingsAccount();
      │              │                      │
      │              │                      │
      │              │                      └── Object Type
      │              └──────────────────────── Reference Variable
      └─────────────────────────────────────── Reference Type
```

### 🏷️ Reference Type

```java
BankAccount
```

This tells Java

> "The variable `account` can refer to any object that is a BankAccount."

Examples

```java

account = new CurrentAccount();

account = new SalaryAccount();
```

All are valid because each class extends `BankAccount`.

---

### 📝 Reference Variable

```java
account
```

This variable stores the reference to whichever BankAccount object is currently assigned.

---

### 🎯 Object Type

```java
SavingsAccount
```

This is the actual object created in memory.

Tomorrow, the object type may change.

```java
BankAccount account = new PremiumSavingsAccount();
```

The reference type is still

```java
BankAccount
```

Only the object type changes.

# 📊 Summary

| Concept               | Meaning                            | Example                        |
| --------------------- | ---------------------------------- | ------------------------------ |
| 🏗️ Class             | Blueprint                          | `class Car {}`                 |
| 📦 Object             | Real instance created from a class | `new Car()`                    |
| 📍 Reference          | Address of an object               | Returned by `new`              |

# 🎤 Interview Questions

### Q1. What is a reference?

**Answer:**

A reference is a value that points to the memory location of an object. It allows Java programs to access objects stored in memory.

---

### Q2. What is a reference variable?

**Answer:**

A reference variable is a variable that stores the reference (address) of an object rather than the object itself.

Example

```java
Car car = new Car();
```

Here,

```java
car
```

is the reference variable.

---

### Q3. What is a reference type?

**Answer:**

The reference type is the declared type of a reference variable. It determines what type of objects the variable can refer to.

---

### Q4. What is an object type?

**Answer:**

The object type is the actual class of the object created using the `new` keyword.

Example

```java
Animal animal = new Dog();
```

Reference Type → `Animal`

Object Type → `Dog`

---

# 📝 Key Takeaways

✅ A class is a blueprint.

✅ An object is a real instance created from a class.

✅ The `new` keyword creates an object in memory.

✅ A reference points to the memory location of an object.

✅ A reference variable stores that reference.

✅ The reference type determines what kinds of objects a variable can refer to.

✅ The object type is the actual class of the object created using `new`.

---

# ⚙️ Compile Time vs Runtime

Before discussing reference types, let's understand two important phases of a Java program.

## 📖 Compile Time

### Definition

> Compile time is the phase in which the Java compiler checks your source code for syntax errors and type safety before the program runs.

During compile time,

the compiler checks things like

* Is the syntax correct?
* Are variables declared?
* Are methods available?
* Are data types compatible?

If any error is found,

the program will **not run**.

---

## 📖 Runtime

### Definition

> Runtime is the phase in which the compiled program actually executes.

During runtime,

Java creates objects,

allocates memory,

calls methods,

and executes your program.

---

## 📊 Visual Representation

```text
Java Source Code
        │
        ▼

     Compile Time
-------------------------

✔ Type Checking

✔ Method Availability

✔ Compilation
        │
        ▼
.class File
        │
        ▼
-------------------------
      Runtime
-------------------------
✔ Objects Created

✔ Methods Executed

✔ Output Produced
```

---

# 🌟 The Golden Rule

There are two different questions Java asks.

## 🧠 Compile Time Question

> **"Can this method be called?"**

Compiler answers this using

```text
Reference Type
```

---

## 🚀 Runtime Question

> **"Which implementation of this method should execute?"**

Java answers this using

```text
Object Type
```

This single idea is the foundation of runtime polymorphism.

---

# 🎯 Why Does the Reference Type Control Accessible Methods?

This is probably the most important concept in Java OOP.

Consider these classes.

```java
class Animal{

    void eat(){

        System.out.println("Animal is eating");

    }

}
```

```java
class Dog extends Animal{

    void bark(){

        System.out.println("Dog is barking");

    }

}
```

Notice carefully.

`Dog` has

* `eat()` (inherited)
* `bark()` (its own method)

---

Now create the object.

```java
Animal animal = new Dog();
```

Let's identify everything.

```text
Animal      animal      =      new Dog();
   │             │                 │
   │             │                 │
   │             │                 └── Object Type
   │             └──────────────────── Reference Variable
   └────────────────────────────────── Reference Type
```

---

# 🔍 What Happens When the Compiler Sees

```java
animal.eat();
```

The compiler does **not** immediately look at

```text
Dog
```

Instead,

it first checks the **reference type**.

```text
Reference Type

↓

Animal
```

Compiler asks

> Does `Animal` have a method named `eat()`?

Answer

```text
Yes
```

Compilation succeeds.

---

Now consider

```java
animal.bark();
```

Again,

the compiler checks

```text
Reference Type

↓

Animal
```

Question

> Does `Animal` have `bark()`?

Answer

```text
No
```

Compiler immediately reports

```text
Cannot find symbol

bark()
```

The program never runs.

---

# ⚠ Important Observation

Notice something very important.

The compiler never checks

```text
Dog
```

It only checks

```text
Animal
```

because the variable is declared as

```java
Animal animal;
```

This is called **compile-time type checking**.

---

# 🎨 Visual Representation

```text

↓

✔ eat()

❌ bark()

NOT HERE

Dog
```

# 🤔 Why Doesn't the Compiler Check the Object?

Many beginners ask

> "The object is Dog, so why doesn't Java simply check Dog?"

The answer is simple.

Because at compile time,

Java cannot always know which object will be assigned.

Consider

```java
Animal animal;
```

At this point,

no object has been assigned.

Later,

someone may write

```java
animal = new Dog();
```

or

```java
animal = new Cat();
```

or

```java
animal = new Lion();
```

All are valid.

Now imagine Java allowed

```java
animal.bark();
```

What happens if later

```java
animal = new Cat();
```

Should Java execute

```java
animal.bark();
```

Cats don't bark.

That would be impossible.

Therefore,

Java says

> I will only allow methods that **every Animal is guaranteed to have.**

That guarantee comes from the **reference type**.

# 🎤 Interview Questions

### Q1. Why does the reference type control accessible methods?

**Answer:**

The compiler performs compile-time type checking using the reference type. It verifies that every method call is declared in the reference type so that the code remains valid regardless of which subclass object is assigned at runtime.

---

### Q2. Why doesn't the compiler check the object type?

**Answer:**

At compile time, the compiler cannot always know which subclass object will be assigned to a reference. Therefore, it checks only the declared reference type to ensure type safety.

---

### Q3. What determines which overridden method executes?

**Answer:**

The object's actual type determines which overridden method executes. Java performs this decision at runtime using Dynamic Method Dispatch.

---

# 📝 Key Takeaways

✅ Compile time and runtime are two different phases.

✅ The compiler checks only the **reference type**.

✅ The compiler asks:

> "Can this method be called?"

✅ Runtime checks the **object type**.

✅ Runtime asks:

> "Which overridden method should execute?"

✅ Compile-time type checking ensures type safety.

✅ Runtime method dispatch provides runtime polymorphism.

# 🌟 Relation with Runtime Polymorphism

## 📖 Definition

> Runtime Polymorphism means that the same reference can refer to different objects, and Java decides at runtime which overridden method to execute.

Consider

```java
abstract class BankAccount {

    abstract void deposit();

}
```

```java
class SavingsAccount extends BankAccount {

    @Override
    void deposit() {

        System.out.println("Savings Deposit");

    }

}
```

```java
class PremiumSavingsAccount extends BankAccount {

    @Override
    void deposit() {

        System.out.println("Premium Deposit");

    }

}
```

Now

```java
BankAccount account = new SavingsAccount();

account.deposit();
```

Output

```text
Savings Deposit
```

Later

```java
account = new PremiumSavingsAccount();

account.deposit();
```

Output

```text
Premium Deposit
```

Notice something.

The method call never changed.

```java
account.deposit();
```

Only the object changed.

This is Runtime Polymorphism.

---

# 🎯 Why is This Powerful?

Without polymorphism,

you would have to write

```java
SavingsAccount account =
        new SavingsAccount();
```

Later

```java
PremiumSavingsAccount account =
        new PremiumSavingsAccount();
```

Now every place using `SavingsAccount` must change.

Instead,

Java allows

```java
BankAccount account =
        new SavingsAccount();
```

Later

```java
BankAccount account =
        new PremiumSavingsAccount();
```

The business logic remains unchanged.

# 🎤 Interview Questions

## Q1. Why does Java separate the reference type from the object type?

### ✅ Answer

Java separates the reference type from the object type to support runtime polymorphism. The reference type provides compile-time type safety, while the object type allows different implementations to be substituted and their overridden methods executed at runtime.

---

## Q2. Why is

```java
BankAccount account = new SavingsAccount();
```

preferred over

```java
SavingsAccount account = new SavingsAccount();
```

### ✅ Answer

Using the parent type (`BankAccount`) makes the client depend on an abstraction rather than a concrete implementation. This enables runtime polymorphism and reduces coupling, making it easier to replace implementations.

---

## Q3. What is runtime polymorphism?

### ✅ Answer

Runtime polymorphism is the ability of a parent reference to refer to different child objects, with Java selecting the appropriate overridden method at runtime based on the object's actual type.

---

## Q4. How does this help achieve loose coupling?

### ✅ Answer

The client depends on an abstraction instead of a concrete class. Different implementations can be substituted without changing the client code, improving flexibility and maintainability.

# 🤔 Question 1

## Why doesn't Java always use the object type?

Consider

```java
Animal animal = new Dog();
```

Suppose Java ignored the reference type and always looked at the object.

Then this would work.

```java
animal.bark();
```

Now imagine another developer writes

```java
animal = new Cat();
```

Should this still work?

```java
animal.bark();
```

No.

Cats don't bark.

If Java allowed this, programs would become unpredictable.

To prevent this, Java checks only the reference type during compilation.

This guarantees that every allowed method exists regardless of which subclass object is assigned later.

---

# 🤔 Question 2

## Then what is the purpose of the object type?

The object type has one responsibility.

> **Choose which overridden method should execute.**

Example

```java
class Animal {

    void sound() {
        System.out.println("Animal Sound");
    }

}
```

```java
class Dog extends Animal {

    @Override
    void sound() {
        System.out.println("Dog Barks");
    }

}
```

```java
Animal animal = new Dog();

animal.sound();
```

Compile Time

Compiler asks

```
Does Animal have sound()?
```

Yes.

Compilation succeeds.

Runtime

Java asks

```
Which sound() belongs to the actual object?
```

Answer

```
Dog.sound()
```

Output

```
Dog Barks
```

# 🎤 Interview Questions

## Q1. What is the difference between reference type and object type?

### Answer

The reference type is the declared type of the variable. It determines what objects can be assigned to the variable and what methods are accessible during compilation.

The object type is the actual class of the created object. It determines which overridden method executes at runtime.

---

## Q2. Why is

```java
Animal animal = new Dog();
```

valid?

### Answer

Because Dog inherits from Animal. A Dog object is an Animal, so it can be referenced using an Animal reference.

---

## Q3. Why is

```java
Dog dog = new Animal();
```

invalid?

### Answer

Because every Animal is not necessarily a Dog. Java prevents assigning a parent object to a child reference.

---

## Q4. Who checks accessible methods?

### Answer

The Java compiler checks accessible methods using the reference type.

---

## Q5. Who decides which overridden method executes?

### Answer

The JVM decides at runtime based on the object's actual type.
