# 🤔 What is an Abstract Class?

Before learning the Java definition, let's understand the word **Abstract**.

The word **Abstract** means:

> **Something that represents an idea or concept without providing all of the details.**

The same idea is used in Java.

An abstract class represents a **general concept**.

It tells us:

* What operations are available.
* What every child class must support.

But it does **not** necessarily explain how every operation works.

---

# 📖 Definition

> **An Abstract Class is a class that cannot be instantiated and is designed to be inherited by other classes. It may contain both abstract methods (without implementation) and concrete methods (with implementation).**

Let's understand every part of this definition.

---

# 🔹 "It is a Class"

An abstract class is still a **class**.

That means it can have:

✅ Variables

✅ Constructors

✅ Concrete methods

✅ Static methods

✅ Final methods

✅ Nested classes

The only difference is that it is declared using the `abstract` keyword.

Example:

```java
abstract class BankAccount{

}
```

Even though it is abstract,

it is still a normal Java class with some additional rules.

---

# 🔹 "It Cannot Be Instantiated"

Consider this class.

```java
abstract class BankAccount{

    abstract void deposit(double amount);

}
```

Can we write

```java
BankAccount account =
        new BankAccount();
```

❌ No.

The compiler immediately produces an error.

---

# 🤔 Why?

Imagine Java allowed this.

```java
BankAccount account =
        new BankAccount();
```

Now suppose we call

```java
account.deposit(5000);
```

Which code should Java execute?

The method

```java
deposit()
```

has **no implementation**.

There is no body.

```java
abstract void deposit(double amount);
```

Java has no instructions to execute.

Therefore,

creating an object of an abstract class would create an incomplete object.

Java prevents this by prohibiting object creation.

---

# 🌍 Real-Life Analogy

Imagine an architect creates a blueprint for a building.

The blueprint specifies:

* Number of floors
* Number of rooms
* Position of doors
* Position of windows

Can you live inside the blueprint?

❌ No.

The blueprint is only a design.

A real building must first be constructed.

Similarly,

an abstract class is a blueprint.

Child classes are the actual buildings.

```text
Blueprint

↓

Apartment Building

↓

People Live Here
```

The blueprint itself is never used directly.

---

# 🏦 Banking Example

Suppose we are developing banking software.

Every account should support:

* Deposit
* Withdraw
* Check Balance

But every account type performs these operations differently.

So we write

```java
abstract class BankAccount{

    abstract void deposit(double amount);

    abstract void withdraw(double amount);

}
```

This class describes

**what every bank account should do.**

It does **not** explain

**how**.

---

# 💻 Concrete Implementation

Savings Account

```java
class SavingsAccount extends BankAccount{

    @Override
    void deposit(double amount){

        System.out.println("Savings Deposit");

    }

    @Override
    void withdraw(double amount){

        System.out.println("Savings Withdrawal");

    }

}
```

Current Account

```java
class CurrentAccount extends BankAccount{

    @Override
    void deposit(double amount){

        System.out.println("Current Deposit");

    }

    @Override
    void withdraw(double amount){

        System.out.println("Current Withdrawal");

    }

}
```

Now Java knows exactly which implementation to execute.

---

# 🔍 Why Didn't We Create Only Normal Classes?

A beginner might ask:

> "Why not remove `BankAccount` completely and create only `SavingsAccount` and `CurrentAccount`?"

That is possible for a very small application.

However, in large software systems, we need a **common parent** that represents all bank accounts.

Without it,

every service would depend on specific implementations.

With an abstract class,

every service can communicate using

```java
BankAccount account;
```

instead of

```java
SavingsAccount account;
```

or

```java
CurrentAccount account;
```

This reduces coupling and enables runtime polymorphism.

---

# 🎯 Abstract Class vs Normal Class

| Normal Class                      | Abstract Class                                 |
| --------------------------------- | ---------------------------------------------- |
| Can create objects                | Cannot create objects                          |
| Can be instantiated               | Must be extended                               |
| May contain only concrete methods | Can contain both abstract and concrete methods |
| Represents a complete object      | Represents a general concept or blueprint      |

# 💡 Key Takeaways

✅ An abstract class is still a class.

✅ It cannot be instantiated because it may contain incomplete methods.

✅ It serves as a blueprint for child classes.

✅ It defines common behavior while allowing subclasses to provide specific implementations.

✅ Abstract classes are a practical way to implement abstraction in Java.

---

# 🎤 Interview Questions

### Q1. What is an abstract class?

**Answer:**

An abstract class is a class declared with the `abstract` keyword. It cannot be instantiated and is intended to be extended by subclasses. It may contain both abstract and concrete methods.

---

### Q2. Why can't we create an object of an abstract class?

**Answer:**

Because an abstract class may contain abstract methods without implementations. Creating an object would result in an incomplete object with methods that Java cannot execute.

---

### Q3. Can an abstract class contain concrete methods?

**Answer:**

Yes. An abstract class can contain both concrete methods (with implementation) and abstract methods (without implementation).
